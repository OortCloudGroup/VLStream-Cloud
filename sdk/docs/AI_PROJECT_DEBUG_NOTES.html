4. 长时间卡点：画面没有检测框

现象

已定位
  • 推流正常，但网页画面完全没有框。
  • 日志中推理函数可能返回成功，但检测数量长期为 0。
  • 放置手机等物体无法判断究竟是模型不识别，还是输入/输出处理错误。

为什么这个问题会卡很久

“没有框”不是单一问题，它至少可能发生在以下任一层：

  1. 模型文件与芯片不匹配，模型根本没有正确执行。
  2. 输入格式错误，例如 RGB/BGR、NCHW/NHWC、uint8/float、stride 或 letterbox 不匹配。
  3. 辅助输入没有创建，模型执行结果无效。
  4. 输出布局理解错误，把坐标、类别、置信度的行列读反。
  5. 坐标映射错误，框被画到画面外。
  6. 阈值过高，所有候选框都被过滤。
  7. 模型本身不包含测试物体对应类别。

最终采用的排查方法

不要继续在实时推流里同时猜模型、预处理、输出解析和画框。先使用 SDK 已知正确的：

  • yolov8.om
  • dog_bike_car_yolov8.bin
  • 独立程序 /mnt/ai_yolov8_verify

离线程序成功打印有效检测框后，证明 ACL 初始化、模型加载、输入组织、模型执行和输出解析都能够工作。之后再逐层接入摄像头帧、坐标还原、YUV 画框和编码推流。

关键经验：模型不确定时，必须先建立“已知模型 + 已知输入 + 已知预期输出”的最小闭环。实时画面无框不能直接得出“模型坏了”的结论。

5. YOLOv8 离线验证与输入尺寸问题

第一次失败：file too large

[VERIFY] input[0] dims=1x3x640x640 type=uint8 size=614400 stride=640
[VERIFY][ERROR] file too large: file=1228800 buffer=614400
[VERIFY] done ret=1

问题不是模型已经执行失败，而是验证程序在执行前就错误地假设“输入文件大小必须小于模型报告的 dataset size”。该模型的 dims、stride、dataset size 和测试 bin 文件之间存在平台特定的组织方式，不能简单整文件 memcpy。

修复方法

验证程序改为依据模型描述中的维度、数据类型、末维行宽和默认 stride，逐行读取输入并分配实际需要的内存；同时 dataset 仍使用 SDK 要求的逻辑大小。

line_size = last_dim * data_type_size;
loop_times = product(other_dims);
alloc_size = loop_times * stride;

for each line:
    read line_size bytes into buffer + line_index * stride

成功标准

[VERIFY] execute ok
[VERIFY] total_valid=3
[VERIFY] box[...] class=... score=... x1=... y1=... x2=... y2=...
[VERIFY] done ret=0

done ret=0 且能打印合理的类别、置信度和坐标，才说明离线推理完整跑通。只有 model loaded 或 execute ok 还不足以证明后处理正确。

模型用途：当前标签是 COCO 80 类。测试文件名 dog_bike_car_yolov8.bin 描述的是测试画面内容，不表示模型只能识别 dog、bicycle 和 car。手机对应 COCO 的 cell phone，但小目标、角度、光照和阈值都会影响结果。

板端验证步骤

cd /ko
sh load3519dv500 -i -sensor0 imx335 -sensor1 imx335
/mnt/ai_yolov8_verify

insmod: can't insert 'sys_config.ko': File exists 通常表示模块已加载，不是本次 AI 验证失败的直接原因。

6. YOLO11 转换和 9 路输出实验

当前状态：YOLO11 已完成离线验证，但因实时链路卡顿而退出主方案。当前正式实时模型仍是 YOLOv8。

RKNN 不能直接变成 OM

参考项目使用的 yolo11n-rk3588-int8.rknn 是 Rockchip RK3588 的编译模型，Hi3519DV500 不能加载。正确路线是找到原始 ONNX/训练模型，再通过 HiSilicon 工具链生成 OM；不能只修改后缀，也不能可靠地把 RKNN 反向转换成 OM。

模型加载失败：缺少 libsvp_aicpu.so

Error loading shared library libsvp_aicpu.so: No such file or directory
svp_acl_mdl_load_from_mem failed ret=100003

该 OM 包含需要 AICPU 支持的算子。解决方式是把 SDK 对应架构和 libc 的 libsvp_aicpu.so 打入 rootfs，并确保动态链接器搜索路径正确。库缺失时模型甚至无法创建，和检测阈值无关。

模型不是单输入：3 输入

input[0] name=images   dims=1x3x640x640       size=1228800
input[1] name=task_buf dims=1x1x1x108         size=432
input[2] name=work_buf dims=1x1x1x1731840     size=6927360

初版验证程序只创建 images 一个输入，因此报 expects one input, model inputs=3 dataset=1。修复后为 task_buf 和 work_buf 分配 SDK 要求的缓冲区并清零，模型才能执行。

模型输出不是最终 300 个框：9 路特征图

分支网格步长输出内容
020x203264 通道 DFL 框 + 80 类分数 + 1 路 score sum
140x401664 通道 DFL 框 + 80 类分数 + 1 路 score sum
280x80864 通道 DFL 框 + 80 类分数 + 1 路 score sum

后处理需要对每个网格执行 DFL softmax/期望值解码、类别筛选、坐标还原，再做 class-aware NMS。离线成功结果包含：

class=16 dog      score=0.945312
class=1  bicycle  score=0.937500
class=14 bird     score=0.929688
class=2  car      score=0.906250
[VERIFY] done ret=0

为什么移植实时后很卡

  • 9 路输出总数据量明显高于 YOLOv8 当前的“数量 + 6x300 框”输出。
  • work_buf 本身接近 7 MB，每次数据集和缓存管理压力更大。
  • DFL、三尺度遍历和 NMS 增加 CPU 后处理开销。
  • 模型切换成功不等于实时性能满足 15 fps 推流要求。
决策：YOLO11 实验保留为验证资产，但实时链路退回已验证且输出更轻的 yolov8.om。

7. 长时间卡点：画面卡顿、延迟和残影

早期根因：推理阻塞视频处理

如果取帧、预处理、ACL execute、后处理、画框和编码全部在同一线程串行执行，一次 50 ms 左右的推理就会直接叠加到视频帧周期。再加上取帧、复制和编码，浏览器看到的就是低帧率和延迟堆积。

当前解决方法：真正的异步线程

  • 视频线程持续从 VPSS 取帧、叠加缓存框并送 VENC。
  • 每 10 帧才向 AI 队列提交一次，减少 NPU 和内存带宽压力。
  • AI 队列是单槽“最新帧”设计；旧任务尚未处理时，新帧覆盖旧帧，避免越积越延迟。
  • AI 线程完成推理后更新共享检测结果，视频线程读取缓存结果，不等待本次推理完成。

如何读性能日志

[INFO] AI send profile frame=690 get=51ms mmap=0ms queue=17ms
       overlay=0ms venc=1ms total=69ms det=0
[AI] profile frame=690 create_in=8ms preprocess=31ms execute=14ms
     parse=0ms draw=0ms total=53ms det=0
字段含义异常判断
get等待 VPSS 帧接近帧周期通常正常，不等于 CPU 忙了 51 ms
queue复制并提交 AI 帧持续升高说明 memcpy 或锁竞争明显
preprocessYUV/RGB、缩放、letterbox、布局转换当前约 30 ms，是主要 CPU 成本之一
executeNPU 模型执行当前 YOLOv8 常见约 14-15 ms
overlay绘制缓存框和文字通常接近 0 ms
venc送入硬件编码器持续高于 80 ms 需要查编码器拥塞

残影不一定来自 AI

运动物体残影还可能来自传感器曝光过长、3DNR、编码码率/GOP、浏览器缓冲或 WebRTC/RTSP 播放策略。曾参考 /home/oort/Desktop/sony_imx335 尝试修改 sensor 寄存器，但用户随后要求撤销，因此当前不能把传感器参数变化当成已采用的解决方案。

排查残影时先做 A/B：关闭 AI 但保持相同编码和播放端。如果残影仍存在，优先查 ISP/曝光/降噪/编码；如果仅开启 AI 后出现，才继续查帧复制、排队和推理阻塞。

8. 框闪烁、框不准与文字绘制

为什么框会一闪而过

每 10 帧才推理一次时，如果只在推理成功的那一帧画框，其余 9 帧都没有框，肉眼就会看到闪烁。当前实现缓存最近一次检测结果，并允许它在后续 30 帧继续绘制；连续若干次空检测后再清除，兼顾稳定显示和目标离开后的及时消失。

为什么置信度调到 0.25 后更容易误检

0.25 的目标是先保证链路有结果，但低置信候选更多。例如把背景误判为 umbrella、bottle。它不代表后处理一定错误，必须结合以下因素判断：

  • 同一目标在连续多次推理中类别和位置是否稳定。
  • 框坐标是否与物体一致，是否存在整体缩放/偏移。
  • 输入颜色顺序、归一化和 letterbox 是否与模型转换配置一致。
  • classes.txt 顺序是否与模型训练类别顺序一致。
  • 目标是否太小、逆光、模糊或被遮挡。

当前绘制内容

画框直接作用于 YVU420SP 帧，文字开关为 AI_DRAW_LABEL_TEXT = true，标签包含类别名和置信度。上报线程拿到的是画框后的同一帧，因此事件图片应当包含框和文字。

如果框整体位置正确但类别经常错误，优先查模型精度和阈值；如果框整体按固定比例偏移，优先查 letterbox 与坐标逆变换；如果所有分数都异常接近 0，优先查输入预处理和输出数据类型。

9. 事件上报：已成功 POST，但平台仍不可见

当前实现

  • 独立模块：http_reporter.cpp/.h。
  • 推流线程只复制事件数据并入队，不执行网络请求。
  • 队列最大 4 个，满时丢最旧任务，避免网络故障拖垮内存。
  • 启动前 30 帧不提交；两次提交至少间隔 10000 ms。
  • YVU420SP 转 RGB，再用 stb 编码为质量 80 的 JPEG。
  • HTTP 200 才算成功；响应存在 success 字段时还必须为 true。

multipart/form-data 的实际结构

POST /vlsEventManagement/report HTTP/1.1
Host: 192.168.60.76:32557
Content-Type: multipart/form-data; boundary=----rtsp-streamer-event-boundary-7f3d9a01

------rtsp-streamer-event-boundary-7f3d9a01
Content-Disposition: form-data; name="event"
Content-Type: application/json

{event JSON}
------rtsp-streamer-event-boundary-7f3d9a01

等价 cURL

curl --location 'http://192.168.60.76:32557/vlsEventManagement/report' \

  --form 'file=@/path/frame_780.jpg;type=image/jpeg;filename=frame_780.jpg'

成功日志说明什么

[REPORT] url=http://192.168.60.76:32557/vlsEventManagement/report
targets=1 jpeg=99413 http=200
resp="{"code":200,"success":true,"data":true,"msg":"操作成功"}"

它证明设备已连接到该地址、完成上传并收到后端成功响应。它不能证明 VLStream 页面查询的就是同一数据库、租户或项目。

平台页面没有新记录时的后端检查项

  1. 192.168.60.76:32557 是否与 workup-dev 页面连接的是同一套后端环境。
  2. /report 最终写入哪张表，是否发生事务回滚或异步消费失败。
  3. 平台列表是否按租户、组织、设备、删除状态、事件状态或时间范围过滤。
  4. reportDevice=AETY-00-NJN2-WJUB-00000100 是否能关联平台设备表。
  5. 板端时间是否正确；错误的未来/过去时间可能让记录落在页面筛选范围外。

为什么不再调用 /save

曾临时同时调用 /report 和 /save，日志显示 /report 返回 200，而 /save 返回 401“请求未授权”。/save 是需要平台鉴权的管理接口，不是设备匿名上报前必须调用的步骤。该双上报代码已按要求撤销。

Apifox 的 415 错误

{
  "code": 415,
  "success": false,
  "msg": "Content-Type 'application/octet-stream' is not supported"
}

当时选择的是 2.png，Apifox 以 application/octet-stream 上传。解决方法：

  • 使用真实 JPEG 文件，不能只把 PNG 扩展名改成 JPG。
  • event 子项设置 Content-Type: application/json。
  • file 子项设置 Content-Type: image/jpeg。
  • 不要手工覆盖整个请求的 multipart Content-Type，让 Apifox 自动生成 boundary。

  • 入口声称支持 HTTPS，但下载函数实际只实现 HTTP。
  • 直接写正式模型文件，没有 .tmp + rename。
  • 先返回成功再调用切换回调，并忽略回调返回值。
  • 固定使用 RKNN/ONNX，与 Hi3519 的 OM/ACL 不兼容。

当前仍需改进的风险

当前实现先验证 .tmp，随后 rename 覆盖正式路径，再从正式路径重新加载。如果第二次加载意外失败，旧内存上下文仍可继续推理，但磁盘上的正式模型已经被替换，重启后可能无法恢复。更稳妥的方案是版本化文件、候选上下文一次加载、原子交换 active 指针，并保留上一版本用于回滚。
  • 当前下载器只真正支持 http://，不支持 HTTPS、重定向、chunked 和鉴权 URL。
  • 仅“模型可加载”不够，还应校验输入尺寸、输入数量、输出数量和后处理类型。
  • 热更新到不同类别模型时，必须同步 labels 和后处理配置。

为什么网页平台可能无法直接下发

#/algorithm-arrangement 是前端路由。真正发请求的应是 VLStream 后端。设备地址 192.168.88.98 是私网地址，如果云端后端没有路由/VPN/网关，就无法访问设备的 8888 端口。浏览器从 HTTPS 页面直接访问设备 HTTP 还会受到 mixed-content 和 CORS 限制。

# 点击平台下发时，在板端观察是否有连接
tcpdump -ni eth0 tcp port 8888

完全没有 SYN 包说明问题在平台调用或网络可达性，不在模型加载代码。

11. IP、DHCP、MAC 与设备标识

自动选网口脚本的含义

脚本依次检查 eth0 eth1 end0 的 carrier，找到插线网口后启动接口并尝试 DHCP；只有获得 IPv4 地址才认为成功。所有 DHCP 都失败时，再给 eth0 设置静态地址。

为什么同一根网线重启后 IP 仍会变化

DHCP 地址由路由器租约决定，不是由网线决定。正常情况下，同一稳定 MAC 常会续租同一地址，但没有绝对保证。项目排查中还发现每次启动显示的 MAC 不同，例如：

eth0 Link encap:Ethernet HWaddr E6:44:5E:F6:99:C4

如果固件没有从 OTP、EEPROM 或持久化分区读取固定 MAC，驱动可能生成随机本地 MAC。路由器会把它视为一台新设备，因此分配不同 IP。

手工 DHCP 成功说明什么

udhcpc -i eth0 -s /etc/udhcpc.script -q -n -t 3 -T 2
udhcpc: lease of 192.168.88.48 obtained from 192.168.88.1

说明网线、carrier、DHCP 服务器和 udhcpc 客户端基本正常。此前回退到 192.168.88.98，并不表示 DHCP 不可用，可能是启动时 DHCP 超时、脚本不完整或 MAC 变化导致租约变化。

设备 ID、IP、MAC 不是同一个字段

当前决策与量产建议

当前联调阶段先固定 192.168.88.98/24，网关 192.168.88.1，避免调试地址反复变化。量产时更合理的是：

  1. 每台设备从 OTP/EEPROM/独立持久化分区获得唯一且稳定的 MAC。
  2. 使用 DHCP，由路由器按 MAC 保留地址；或者平台通过设备主动长连接，不依赖反向访问私网 IP。
  3. 不同设备不能烧录完全相同的固定 MAC，也不应全部固定为同一个 IP。

为什么每次改代码后必须确认镜像内容

多次出现“代码已经改了，但板端行为像上一个版本”的疑问。常见原因包括：

  • 只编译了应用，没有把新二进制复制进 rootfs staging 目录。
  • rootfs 镜像没有重新生成，或烧录了旧路径下的镜像。
  • 板端同时存在多个同名程序，实际运行的不是新文件。
  • 模型、配置或动态库没有一起打包。

每次构建后的核对清单

  • 检查构建命令退出码为 0。
  • 检查 ext4 镜像修改时间和大小。
  • 记录镜像 MD5，确保烧录端拿到同一文件。
  • 确认 rootfs 中包含新的 rtsp_streamer、OM、classes.txt 和配置。
  • 若模型依赖 AICPU，确认 libsvp_aicpu.so 已打包且架构正确。
  • 烧录后从启动日志确认模型路径、配置和功能开关。

13. 常见日志的正确判读

日志含义下一步
sys_config.ko: File exists内核模块已加载不是 AI 失败证据；避免重复加载即可
model loaded ... inputs=4 outputs=2模型描述创建成功继续看 execute 和检测结果，不能到此就宣布推理成功
execute okNPU 执行返回成功继续验证输出解析和框坐标
done ret=0 且有正确框离线闭环成功可以移植同一输入/输出逻辑到实时链路
detections=0本次没有通过阈值的结果检查目标、阈值、预处理，不等于模型一定坏了
overwritten=0AI 单槽队列暂未覆盖任务如果持续增长，说明 AI 处理赶不上提交频率
http=200 success=true后端接受了事件请求平台不可见时查数据库环境、租户和列表过滤
/save http=401管理接口需要鉴权设备端继续使用 /report，不应先调 /save
415 application/octet-stream上传文件 MIME 错误使用真实 JPEG，file 子项设置 image/jpeg
load new model ok候选模型可被 ACL 初始化继续确认 switched 日志和切换后推理结果
crng init done内核随机数初始化完成正常启动信息，通常无需关注

设备自动崩溃/断电的判断边界

应用日志突然停止不能单独证明应用导致“断电”。需要区分：

  • 应用进程崩溃：shell 仍可操作，可能有 segmentation fault。
  • 内核崩溃：串口通常出现 panic、oops、watchdog 或调用栈。
  • 电源/硬件复位：串口直接中断或重新出现 bootrom/U-Boot 启动。
  • 内存压力：可能出现 OOM killer、分配失败或驱动异常。

曾有固件运行一段时间后设备异常停止，但仅凭应用检测日志无法确定电源级根因。降低推理频率、使用已验证 YOLOv8、减少大缓冲和异步解耦改善了稳定性；若再次出现，必须保留异常前后的完整串口和内核日志。

识别精度

需要数据验证

当前 0.25 阈值用于先跑通，误检仍明显。应收集真实摄像头样本，逐项核对预处理，再基于 precision/recall 调阈值，必要时使用面向目标场景训练和转换的模型。

量产网络身份

需要硬件/生产方案

每台设备必须有稳定唯一 MAC 和业务 serno。固定 IP 只适合当前单板联调，不能直接复制到多设备部署。

新问题的标准排查顺序

  1. 先保存完整日志和固件 MD5，确认运行版本。
  2. 把问题分到采集、推理、后处理、绘制、编码、网络或平台数据库中的一层。
  3. 使用离线程序或 cURL/Apifox 建立最小可复现闭环。
  4. 一次只改变一个变量，并记录修改前后的性能和输出。
  5. 实验方案不采用时明确撤销，避免后续把旧实验误认为当前配置。
